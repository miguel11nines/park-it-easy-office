## New Findings — Still Open (7)

Discovered during the v2.4.1 re-audit. 5 of the original 12 new findings were resolved in v2.5.0 (see Phase 3A Resolved section above). The remaining 7 are listed below.

### New — Medium Severity (2)

#### NEW-M1 — CSP Includes `unsafe-inline` for script-src

| Field        | Detail                                 |
| ------------ | -------------------------------------- |
| **Severity** | **MEDIUM**                             |
| **CWE**      | CWE-693 (Protection Mechanism Failure) |
| **Evidence** | `index.html:7-16`                      |

The CSP meta tag includes `'unsafe-inline'` in `script-src` to support the SPA redirect script in `404.html`. This weakens XSS protection. **Fix:** Externalize the redirect script to a separate `.js` file, or use a hash-based CSP directive.

#### NEW-M3 — Duplicate Booking Check Uses `user_name` Instead of `user_id`

| Field        | Detail                                              |
| ------------ | --------------------------------------------------- |
| **Severity** | **MEDIUM**                                          |
| **CWE**      | CWE-284 (Improper Access Control)                   |
| **Evidence** | `src/components/BookingDialogWithValidation.tsx:74` |

The client-side duplicate booking check in `BookingDialogWithValidation` filters by `user_name` instead of `user_id`. Users with identical display names could see false-positive duplicate warnings, or renamed users could bypass the client-side check. The server-side `UNIQUE(user_id, date)` constraint mitigates this, but the client UX is incorrect. **Fix:** Change the filter to use `user_id`.

### New — Low Severity (5)

#### NEW-L2 — signUp Missing `emailRedirectTo` Option

| Field        | Detail                                                |
| ------------ | ----------------------------------------------------- |
| **Evidence** | `src/pages/Auth.tsx` vs `src/services/authService.ts` |

The `signUp` call in `Auth.tsx` does not set `emailRedirectTo`, while `authService.ts` does. Users confirming email may be redirected to the wrong URL. **Fix:** Set `emailRedirectTo` consistently.

#### NEW-L3 — Sidebar Cookie Missing `Secure; SameSite` Attributes

| Field        | Detail                   |
| ------------ | ------------------------ |
| **Evidence** | Sidebar state management |

Overlaps with L1 from original report but confirmed still present. The sidebar state cookie lacks `Secure` and `SameSite` attributes.

#### NEW-L4 — `waitlist_with_users` View Exposes All Waiting Users

| Field        | Detail                                |
| ------------ | ------------------------------------- |
| **Evidence** | `waitlist_with_users` view definition |

The `waitlist_with_users` view joins waitlist entries with user display names and emails, and is accessible to all authenticated users. An authenticated user can see who else is on the waitlist. **Fix:** Add RLS filtering to restrict to the current user's waitlist entries, or accept as intended behavior if waitlist visibility is a product requirement.

#### NEW-L5 — `UNIQUE(user_id, date)` May Block AM+PM Bookings

| Field        | Detail                                                         |
| ------------ | -------------------------------------------------------------- |
| **Evidence** | `supabase/migrations/20260223000001_phase1_security_fixes.sql` |

The `UNIQUE(user_id, date)` constraint prevents a user from creating two bookings on the same date, even if one is `morning` and the other `afternoon`. If the business intent allows split-day bookings, this constraint is overly restrictive. **Verify:** Confirm business rules — if AM+PM should be allowed, change to `UNIQUE(user_id, date, duration)` or similar.

#### NEW-L7 — Zero Unit Tests for Security-Critical Code

| Field        | Detail                                                                                                     |
| ------------ | ---------------------------------------------------------------------------------------------------------- |
| **Evidence** | `src/lib/errorMessages.ts`, `src/components/ErrorBoundary.tsx`, `src/services/authService.ts`, all 7 hooks |

The `getUserErrorMessage()` function (introduced in v2.4.0), `ErrorBoundary`, `authService.ts`, and all hooks have zero unit test coverage. Security-critical error mapping and auth logic have no automated verification. **Fix:** Add unit tests for `getUserErrorMessage()`, auth flows, and error boundary behavior.
